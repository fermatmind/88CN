# 18 Structured Data Policy

## Date

2026-06-15

## Purpose

Define rules for schema.org structured data markup on 88CN public pages. Structured data improves search engine understanding of page content and enables rich result eligibility without traditional SEO content tactics.

## Supported Schema Types

| Schema Type | Used On | Required Properties |
|---|---|---|
| `WebPage` | Category pages, Founders page, Genesis Badge page, Project detail pages | `url`, `name`, `description` |
| `ItemList` | Category pages, Report list page | `itemListElement[]` with `ListItem` entries |
| `CollectionPage` | Collection detail pages | `url`, `name`, `description`, `mainEntity` (ItemList) |
| `TechArticle` | Report detail pages | `url`, `headline`, `description`, `datePublished`, `dateModified` |

## Forbidden Markup

The following schema.org properties and types must not be used on 88CN pages:

- `AggregateRating` — No fake ratings. No community ratings system exists in Phase 1.
- `Review` — No fake reviews. No review system exists in Phase 1.
- `Offer` — No offers, pricing, or commercial transactions exist on 88CN.
- `Organization` with fake attributes — Do not fabricate organization data.
- Any property that implies fabricated metrics (fake `interactionStatistic`, fake `member` counts, fake `subscriber` numbers)

## Markup Rules

1. **Mark visible content only.** Do not use structured data to describe content that is not visible to users on the page.

2. **Use JSON-LD only.** JSON-LD is the recommended format by schema.org and search engines. Do not use Microdata or RDFa.

3. **One script tag per schema object.** Each structured data object should be in its own `<script type="application/ld+json">` tag.

4. **URLs must be absolute.** All `url` and `@id` properties must use fully qualified absolute URLs.

5. **Description must match visible content.** The `description` property in structured data must accurately reflect the visible page content. Do not stuff keywords or use descriptions that differ from what users see.

6. **Date properties must be real.** `datePublished` and `dateModified` must reflect actual publication and modification dates. Do not use future dates or fabricated timestamps.

7. **ItemList position must be sequential.** `ListItem` `position` values must start at 1 and increment sequentially without gaps.

## Implementation

Structured data is generated by utility functions in `lib/structured-data.ts`:

- `webPageJSONLD(url, title, description)` — Generates `WebPage` schema
- `itemListJSONLD(items)` — Generates `ItemList` schema from name/url pairs
- `collectionPageJSONLD(url, title, description, items)` — Generates `CollectionPage` schema with embedded `ItemList`
- `techArticleJSONLD(url, headline, description, datePublished)` — Generates `TechArticle` schema

These functions are composable and do not depend on any external data source or database.

## Future Schema Types

The following schema types are candidates for future phases:

- `FAQPage` — If a page has structured FAQ content (category hubs already have FAQ blocks; they could be marked up as FAQPage in a future update)
- `BreadcrumbList` — If breadcrumb navigation is added
- `SoftwareApplication` — For project detail pages, if sufficient structured product data is available

These are not implemented in Phase 1 and should not be added without separate review.

## No Index Policy for Non-Public States

Pages for projects in `submitted`, `pending_review`, `approved` (pre-public), and `archived` states must not be indexed. These pages must:

1. Include `<meta name="robots" content="noindex">` in the HTML `<head>`
2. NOT be included in sitemap output
3. NOT use `robots.txt` `Disallow` as a substitute for `noindex`

`robots.txt Disallow` prevents crawling but does not prevent indexing. A page blocked by `robots.txt` can still appear in search results. Only `noindex` reliably prevents indexing.
