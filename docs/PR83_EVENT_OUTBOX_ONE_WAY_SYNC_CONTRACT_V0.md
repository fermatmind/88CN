# PR83 Event Outbox + One-way Sync Contract v0

## Result

Validation target: PR83 defines a one-way async event outbox contract only.

This task does not create a Supabase migration, Supabase trigger, webhook runtime, webhook credential, Laravel runtime, Redis production config, server config, full-table sync, request-time polling, external service connection, or data repository mutation.

## Contract

Machine-readable contract:

- `ops/contracts/event-outbox.json`

Contract posture:

| Area | Decision |
| --- | --- |
| Flow direction | One-way async event outbox |
| Source of truth | Existing primary reviewed data system |
| Future consumer | Downstream feed service after checkpoint approval |
| Request-time sync | Not allowed |
| Public route side effect | Not allowed |
| Admin route side effect | Not allowed |
| Full-table sync | Not allowed |
| I/O-heavy polling | Not allowed |

## Allowed Event Types

Allowed future event types:

- `project_published`
- `project_archived`
- `project_updated_public_fields`
- `changelog_published`
- `source_link_verified`
- `profile_claimed_public`
- `profile_removed_public`

PR83 only defines these event names. It does not create an event producer or consumer.

## Event Envelope

Required future envelope fields:

| Field | Rule |
| --- | --- |
| `event_id` | Opaque UUID generated by a future outbox producer |
| `event_type` | Must be one of the allowed event types |
| `schema_version` | Contract version for validation |
| `idempotency_key` | Stable per logical public-record change |
| `occurred_at` | UTC event timestamp |
| `public_record_ref` | Public slug or public profile URL only |
| `payload` | Allowlisted public fields only |

## Allowed Payload Fields

Payloads may only include:

- `project_slug`
- `project_name`
- `public_profile_url`
- `canonical_url`
- `category`
- `technology_tags`
- `public_source_links`
- `published_status`
- `public_changelog_summary`
- `machine_readability_summary`
- `last_reviewed_at`

## Denied Payload Fields

Event payloads must not include:

- private founder, submitter, claim, or contact fields;
- admin notes, review notes, review comments, or reviewer identifiers;
- payment or commercial order state;
- API key, customer key, or metering fields;
- raw payloads or raw Supabase rows;
- webhook credentials or other credential material;
- claim secrets;
- private telemetry;
- internal Source Confidence inputs;
- internal Signal Score inputs and hidden score weights.

## Idempotency Policy

Future idempotency key shape:

```text
event_type:public_record_ref:schema_version:public_change_revision
```

Consumer rule:

- duplicate `idempotency_key` values must be treated as already processed.

Producer rule:

- the same logical public-record change must not generate a new idempotency key.

## Retry And Backoff

Future delivery must use bounded retries:

- maximum attempts: 3;
- backoff: exponential with jitter;
- poison events: move to a reviewed error queue;
- failure handling: failed sync must not zero out public scores or remove the last known reviewed public data.

## Data Drift Handling

The existing primary reviewed data system remains the source of truth.

Consumer reconciliation must use:

- `public_record_ref`;
- `schema_version`;
- idempotency key history.

Stale records may be marked stale by a future consumer, but replacement facts must not be invented. Full resync is a manual checkpointed operation only.
